# ch17 피드백 — 게임패드 모드로 변신

## Round 1 (2026-06-18) — fresh 학습자 검증

**판정: needs-work** (17.5 소프트 리셋 방식이 실제로 모드 전환을 못 시킬 가능성 + 라이브러리 식별·보드 인식명·`Keyboard.h` 출처 안내 부족)

### blocker
1. **17.5 `asm volatile ("jmp 0")` 소프트 리셋이 모드 전환을 실제로 못 함** — 본문(188행)은 "보드를 소프트웨어 재시작하면서 EEPROM 플래그를 읽어 그에 맞는 디스크립터로 다시 부팅"한다고 설명하지만, `jmp 0`은 스케치 시작 주소로만 점프할 뿐 ATmega32U4의 USB 모듈을 리셋하지 않는다. 그래서 USB 재열거(re-enumeration)가 일어나지 않아 PC 입장에서는 여전히 이전 디스크립터(키보드 또는 게임패드)로 남는다. fresh 학습자는 "버튼을 2초 눌렀는데 모드가 안 바뀐다"는 결과를 만난다. 게다가 `jmp 0`은 주변장치 레지스터를 초기화하지 않아 동작이 불안정하다. → **물리 리셋(또는 더블탭 리셋 후 재인식)으로 모드가 적용되는 방식**으로 교체하고, 소프트 점프만으로는 USB 디스크립터가 안 바뀌는 한계를 명확히 설명한다.

### major
2. **라이브러리 식별 모호** — `arduino-cli lib install "Joystick"` 결과에 동명(同名) 라이브러리가 여럿 나올 수 있는데, 본문은 어느 것이 ArduinoJoystickLibrary인지 텍스트로 못 박지 않았다. → `arduino-cli lib search joystick` 출력에서 **저자 Matthew Heironimus** 항목을 고르는 법을 본문 텍스트로 명시하고, 동명이 여럿일 때 저자/설명으로 구분하라고 안내한다.
3. **보드명 "Micro"인데 USB 인식명 "Arduino Leonardo"로 뜨는 이유 미설명** — 그림 17.5/17.14 캡션과 17.6 본문은 인식명이 "Arduino Leonardo"라고 하는데, FQBN은 `arduino:avr:micro`다. fresh 학습자는 "내 보드는 Micro인데 왜 Leonardo로 뜨지?" 하고 혼란한다. → ATmega32U4 계열은 USB 식별 정보(VID/PID·제품 문자열)가 Leonardo로 보고되는 경우가 많아 윈도우가 "Arduino Leonardo"로 표시할 수 있다는 한 줄 설명 추가.
4. **`Keyboard.h` 출처 안내 누락** — 17.5 토글 코드가 `#include <Keyboard.h>`, `Keyboard.begin()`, `Keyboard.press/release`를 쓰는데 이 라이브러리가 어디서 왔는지 안내가 없다. → 6장에서 다룬 키보드 라이브러리(5장에서 `arduino-cli lib install "Keyboard"`로 설치)임을 한 줄 안내. `Keyboard.begin()`은 키보드 모드로 부팅할 때 호출한다는 점도 짚는다.

### minor
5. **17.3 축 미설정 안내 부족** — 17.3 기본 예제는 X·Y 축을 `true`로 선언만 하고 `setXAxis/setYAxis`를 호출하지 않아, joy.cpl에서 축 커서가 중앙에 멈춰 있거나 보드별로 0이 아닌 값으로 떠 의아할 수 있다. → 17.3에서는 버튼만 확인하는 단계이고 축은 17.4에서 다룬다는 한 줄 + 필요하면 `setXAxis(0)/setYAxis(0)`로 중립 고정 가능 안내.
6. **핀 충돌 미검증** — 17.5 코드의 LED 핀(9/10/11), 모드 버튼(6), 입력 버튼(2~5)이 앞 장(8장 핀맵)과 충돌하지 않는지 본문이 보장하지 않는다. → 예시 핀이며 자기 키링 핀맵(8장)에 맞춰 조정하라는 한 줄.
7. **Hollow Knight 데모 가정** — 17.7은 "Hollow Knight 데모가 있다면"이라고 특정 게임을 가정하는데 데모 제공은 시기에 따라 바뀐다. → 특정 게임에 의존하지 않도록 "컨트롤러 지원 무료 게임 아무거나"로 일반화하고 Hollow Knight는 예시로만 둔다.

### 적용한 보완 (Round 1)
- 17.5: 소프트 `jmp 0` 리셋을 제거하고 **EEPROM에 모드 저장 후 물리 리셋(또는 USB 재꽂기/더블탭 리셋)으로 재부팅 → 새 디스크립터로 재인식** 방식으로 교체. `jmp 0`만으로는 USB 디스크립터가 안 바뀌어 모드 전환이 실제로 안 된다는 한계를 본문·주의 박스에 명시 [blocker 1].
- 17.2: `arduino-cli lib search joystick` 출력에서 **저자 Matthew Heironimus** 항목을 고르는 법을 본문 텍스트로 명시. 동명이 여럿일 때 저자/설명으로 구분하라고 안내 [major 2].
- 17.6: "Micro 보드인데 인식명이 Arduino Leonardo로 뜨는" 이유(ATmega32U4 계열 USB 식별이 Leonardo로 보고됨)를 한 줄 설명 추가 [major 3].
- 17.5: `Keyboard.h`가 **6장에서 다룬 라이브러리(5장에서 설치)** 임을, `Keyboard.begin()`은 키보드 모드 부팅 시 호출함을 한 줄 안내 [major 4].
- 17.3: 축은 17.4에서 다루고 이 단계는 버튼 확인용임을 명시, 필요 시 `setXAxis(0)/setYAxis(0)` 중립 고정 안내 [minor 5].
- 17.5: LED·버튼 핀은 예시이며 자기 키링 핀맵(8장)에 맞춰 조정하라는 한 줄 [minor 6].
- 17.7: 특정 게임 의존을 줄여 "컨트롤러 지원 무료 게임 아무거나"로 일반화, Hollow Knight는 예시로만 둠 [minor 7].
- 유지: 아이콘 코너, 그림 `<!-- FIG -->` 마커·번호·순서 모두 그대로.

## Round 2 (2026-06-18) — Round 1 정합성 재검

**판정: good** (Round 1 교체분 검증 + 잔여 불일치 2건 수정)

### blocker
없음. 17.5 모드 전환은 이미 **EEPROM 저장 + 물리 리셋(USB 재꽂기/더블탭 리셋)으로 재부팅 → 새 디스크립터 재협상** 방식이라 USB 재인식이 실제로 일어난다(Round 1 교체 정상). `jmp 0`이 USB를 안 바꾼다는 한계도 주의 박스에 남아 있음.

### major
1. **17.5 "보라색 점멸" 서술 ↔ 코드 불일치** — 전환 신호 코드는 `setLED(128,0,128)`을 **1회만 호출**해 보라색을 고정 점등하는데, 코드 주석은 "보라색 점멸"이라 적어 점멸 동작을 암시했다. 본문 절차 설명(298행)은 "LED가 보라색으로 바뀌면"이라 고정 점등을 전제. → 코드가 고정 점등이므로 주석을 "보라색 점등(고정)"으로 수정해 코드·주석·본문을 모두 고정 점등으로 정합화. (점멸을 원하면 별도 토글 코드가 필요하나, 단순함을 위해 고정 점등 유지.)
2. **17.5 `Joystick.h`+`Keyboard.h` 동시 include 시 모드 분리 단정 위험** — 두 라이브러리를 함께 include하고 `begin()` 하나만 호출하는 단일 스케치가 USB 레벨에서 모드를 깔끔히 분리한다고 단정할 수 없다(빌드 시 두 HID 인터페이스가 한 디스크립터에 함께 들어가 "키보드+게임패드 복합 장치"로 잡힐 수 있음; 보드·라이브러리 버전 의존). → 단정 대신 **주의 박스 추가**: `joy.cpl`·메모장으로 직접 검증하라고 안내하고, 확실한 대안으로 **모드별 스케치 2개 분리 업로드**(또는 빌드 플래그로 한 모드만 HID 노출)를 제시. 단일 스케치 예제는 "부팅 모드 선택 흐름" 시연용임을 명시.

### minor
3. **17.3 축 미설정 안내** — 이미 122행에 "X·Y 축을 선언만 하고 값은 17.4에서 채움 + joy.cpl에서 중앙/비중립으로 보일 수 있고 정상 + `setXAxis(0)/setYAxis(0)`로 중립 고정 가능"이 한 줄로 들어가 있음. 추가 수정 불필요(Round 1에서 이미 반영, 본문 17.6 절번호 참조도 정확).

### 적용한 보완 (Round 2)
- 17.5: 전환 신호 LED 주석 "보라색 점멸" → "보라색 점등(고정)"으로 수정. 코드(1회 setLED)·주석·본문 절차를 모두 고정 점등으로 정합화 [major 1].
- 17.5: `Joystick.h`+`Keyboard.h` 동시 include가 모드를 자동 분리한다고 단정하지 않도록 주의 박스 신설. 직접 검증 + 모드별 스케치 분리(가장 확실) 대안 안내 [major 2].
- 17.3 축 미설정 안내는 Round 1 반영분으로 이미 충분 — 변경 없음 [minor 3].
- 유지: 친근체, 아이콘 코너, 그림 `<!-- FIG -->` 마커·번호·순서, 8장 배선 전제 모두 그대로.

## Round 3 (2026-06-18) — 17.5 모드 토글 결함(제목 모순 + 주 경로 재구성)

**판정: needs-work → 수정 완료** (절 제목이 실제 동작과 모순 + 단일 토글 스케치를 주 경로로 제시한 것이 신뢰성 부족)

### blocker
1. **절 제목 ↔ 실제 동작 모순** — 제목 "버튼 길게 누르면 전환"은 버튼만으로 모드가 바뀌는 것처럼 읽히지만, 본문·코드상 버튼은 EEPROM 기록만 할 뿐 실제 전환에는 **물리 리셋이 반드시** 필요하다(USB 디스크립터는 부팅 때만 협상). fresh 학습자는 제목만 보고 "버튼 2초 누르면 끝"으로 오해한다. → 제목을 **"키보드 모드와 게임패드 모드 — 버튼 + 리셋으로 모드 전환"** 으로 정정.

### major
2. **단일 토글 스케치를 주 경로로 제시 — 신뢰성 미보장** — 본문은 이미 (Round 2 주의 박스에서) `Joystick.h`+`Keyboard.h` 동시 include가 모드를 USB 레벨에서 깔끔히 분리한다고 단정 못 함을 인정했는데, 정작 절의 메인 코드는 그 불확실한 단일 토글 스케치였다. 학습자가 가장 먼저 따라 하는 코드가 "보드·라이브러리 버전 따라 결과가 다른" 방식인 것은 부적절. → **확실한 "모드별 스케치 2개 따로 업로드"를 방법 A(권장·주 경로)로 전진 배치**(키보드 스케치/게임패드 스케치를 각각 명확히 제시 + 업로드 명령 분리). 단일 토글 스케치는 **방법 B(심화·실험)** 로 격하하고, 동작이 갈릴 수 있음을 도입부·인용 블록·주의 박스에 명시.

### minor
3. **RGB LED·모드버튼의 8장 배선 전제 미명시** — 방법 B 코드의 RGB LED(9/10/11핀)·모드버튼(6핀)이 8장 배선을 전제한다는 안내가 코드 인라인 주석에만 흩어져 있었다. → 방법 B 코드 바로 앞에 **배선 전제 팁 박스** 신설: LED·모드버튼 핀은 8장 배선 전제이며, RGB LED가 없으면 `setLED(...)`·LED 핀 설정만 지워 LED 색 기능을 생략해도 EEPROM+버튼+리셋 흐름은 그대로 동작함을 명시.

### 적용한 보완 (Round 3)
- 17.5 제목: "버튼 길게 누르면 전환" → **"키보드 모드와 게임패드 모드 — 버튼 + 리셋으로 모드 전환"** [blocker 1].
- 17.5 재구성: **방법 A(권장) = 모드별 스케치 2개 따로 업로드**를 주 경로로 신설(KeyboardMode/GamepadMode 스케치 + 모드별 compile/upload 명령 + "왜 스케치 2개가 더 확실한가" 팁). 기존 단일 토글 스케치는 **방법 B(심화·실험)** 로 격하하고 도입부·인용 경고·주의 박스를 방법 A 우선으로 재배선 [major 2].
- 17.5 방법 B: RGB LED(9/10/11)·모드버튼(6)이 **8장 배선 전제**임을 코드 직전 팁 박스로 명시, LED 없으면 색 기능 생략 가능 안내 [minor 3].
- 17.8 핵심 요약: 모드 전환 항목을 "가장 확실한 건 스케치 2개 따로 업로드(방법 A), 단일 토글(방법 B)은 심화·실험용"으로 갱신.
- 유지: 친근체, 아이콘 코너(배선 전제는 `💡 팁`으로 6종 박스 어휘 준수), 그림 `<!-- FIG -->` 마커·번호·순서(17.7~17.9 그대로), `jmp 0` 소프트 점프 한계 주의 박스, 동명 라이브러리·Leonardo 인식명·데드존 안내 모두 그대로.
