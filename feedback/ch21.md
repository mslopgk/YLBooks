# ch21 피드백 — 깃허브에 공개하고 README 쓰기

## Round 1 (2026-06-18) — fresh 학습자 검증

**판정: needs-work** (공개 한 사이클의 큰 흐름은 좋으나, 저장소 생성 절차가 서로 충돌하고 라이선스 추가 후 동기화·가입 인증·README 편집 위치가 빠져 초보가 따라가다 막힘)

### major
1. **21.3 저장소 생성 흐름 충돌** — 21.3은 `File → New Repository`(빈 폴더에 새 repo 생성)를 안내하면서 동시에 Local path에 "학생이 키링 코드를 저장한 기존 폴더"를 지정하고 `Initialize with README`까지 체크하라고 함. GitHub Desktop의 New Repository는 빈/새 폴더용이고, 기존 코드가 든 폴더를 대상으로 하면 `Add` 흐름이어야 함. 두 절차가 뒤섞여 fresh 학습자는 "기존 코드를 어떻게 올리지?"에서 막힘. → 두 경우를 명확히 분기: (A) 이 책 학생처럼 **이미 키링 코드 폴더가 있는 경우** = `File → Add Local Repository`(없으면 "create a repository" 유도) → 정식 절차로 안내, (B) 새로 시작하는 경우 = `New Repository`로 간단히 보조 안내. README는 한쪽으로 고정.
2. **라이선스 웹 추가 후 로컬-원격 동기화 누락** — 21.7에서 LICENSE 파일을 깃허브 웹에서 만들면 원격에만 커밋이 생기고 로컬은 뒤처진 상태가 됨. 이 상태에서 학생이 GitHub Desktop으로 다시 push하려 하면 충돌/거부됨. 웹에서 편집한 뒤 데스크톱에서 **Pull(Fetch origin → Pull)** 로 내려받아 동기화하라는 안내가 전혀 없음. → 21.7 끝에 Pull 단계 추가.
3. **회원가입 이메일 인증·username 설정 단계 누락** — 21.2 회원가입이 "Sign up 누르고 비밀번호"에서 끝남. 실제 GitHub 가입은 username 입력 + 이메일로 온 인증 코드(launch code) 입력 + 사람 확인(puzzle)을 거쳐야 계정이 활성화됨. 이 단계를 모르면 가입 도중 멈춤. → username·이메일 인증 단계 보강.
4. **README 편집 위치 일관성 부재** — 21.5는 README를 깃허브 웹 에디터(그림 21.10이 "웹의 README 편집 화면")에서 다루는 듯하나, "Claude Code로 초안 받기"는 로컬 폴더 코드를 읽혀 로컬 README.md에 붙여 넣는 흐름. 21.6은 다시 GitHub Desktop에서 commit+push. 웹/로컬을 오가 충돌·혼란 유발. → README는 **로컬 파일(VS Code 등)에서 작성 → 데스크톱 commit+push** 한 방향으로 고정. 그림 21.10 지시문도 로컬 편집기 기준으로 정정.

### minor
5. **git 설치/내장 설명 누락** — 학생이 "git을 따로 설치해야 하나?" 궁금해함. GitHub Desktop이 git을 내장한다는 한 줄 안내 없음. → 21.2에 한 줄 추가.
6. **`docs` 폴더 생성법 부재** — 21.6에서 `docs/demo.gif` 경로를 쓰지만 `docs` 폴더를 어떻게 만드는지 안내 없음. → 폴더 만드는 한 줄 추가.

### 적용한 보완 (Round 1)
- 21.2 회원가입을 username 입력 → 이메일 인증 코드(launch code) 입력 → 사람 확인 순서로 보강. GitHub Desktop이 git을 내장해 별도 설치가 필요 없다는 한 줄 추가.
- 21.3을 "이미 코드 폴더가 있는 경우(Add Local Repository)"를 기본 절차로 재작성. New Repository(처음부터 새로 만드는 경우)는 보조 안내로 분리. README는 로컬에서 다루는 것으로 고정(웹 Initialize 체크 안내 제거).
- 21.5 README 작성을 로컬 파일(VS Code 등) 한 방향으로 고정. 그림 21.10 지시문을 로컬 편집기에서 README.md를 연 화면으로 정정. Claude Code 초안 → 로컬 README.md 저장 → 데스크톱 commit+push 흐름으로 통일.
- 21.6에 `docs` 폴더를 GitHub Desktop의 "show in Explorer/Finder"로 열어 직접 만드는 한 줄 추가.
- 21.7 라이선스 적용 끝에 웹에서 LICENSE를 만든 뒤 GitHub Desktop에서 **Fetch origin → Pull**로 로컬을 동기화하라는 단계와 경고를 추가.
- 그림 마커 id·번호·순서(ch21-f01~f18), 아이콘 코너(note/warning/tip/abstract/question/example/info) 전부 유지. 절 번호 변경 없음.

## Round 2 (2026-06-18) — fresh 학습자 재검증

**판정: needs-work** (공개 사이클 자체는 따라갈 수 있으나, 저장소 이름이 본문 안에서 두 가지로 갈려 URL/예시/연습문제가 어긋나고, README를 21.5에서 쓰고도 공개 반영을 21.6 끝으로 미뤄 GIF를 안 만드는 독자는 README가 공개에 안 올라감)

### major
1. **저장소 이름 불일치** — 21.3은 폴더명 그대로 `keyring-game`으로 저장소가 생성된다고 안내(예시 경로·create a repository 결과·그림 21.5 지시문이 모두 `keyring-game`)하면서, 21.4 확인 URL(`github.com/내아이디/my-keyring-game`)·21.5 5단 구성 제목 예시·Claude Code 프롬프트·연습문제는 `my-keyring-game`을 씀. fresh 학습자가 실제로 만든 `keyring-game`과 본문 후반의 `my-keyring-game`이 달라 URL 접속·README 제목에서 혼란. → 본문 전체를 `my-keyring-game` 한 이름으로 통일.
2. **README 공개 반영 단계 누락** — 21.5에서 README를 작성·저장한 직후 commit + push하는 명시적 단계가 없고 "올리는 건 21.6 끝에서"로 미뤄짐. GIF를 안 만드는 독자는 21.6을 건너뛰어 README가 영영 공개에 안 올라감. → 21.5 끝에 "작성 후 commit + push" 단계를 명시.

### minor
3. **README.md 파일 생성법·Publish Name 필드 부재** — `README.md` 파일을 어떻게 만드는지(VS Code 새 파일, 윈도우 탐색기 확장자 숨김으로 `.md.txt` 되는 함정) 한 줄 없음. Publish 다이얼로그에 저장소 이름 Name 필드가 있다는 언급 없음. → 둘 다 한 줄씩 보강.

### 적용한 보완 (Round 2)
- 21.3: 예시 경로·create a repository 결과·그림 21.5 지시문을 모두 `my-keyring-game`으로 통일. "폴더 이름을 my-keyring-game으로 맞추라 / 다르면 본인 폴더명으로 바꿔 읽으라"는 안내로 합치고, 중복되던 별도 이름 변경 블록쿼트는 제거.
- 21.5: README 작성 뒤 곧장 commit(`add README`) + push로 공개에 반영하는 "### 작성했으면 바로 올리기" 소절 추가. GIF 미작업 독자도 README가 올라가도록 명시. 21.6 미루기 문구 삭제.
- 21.5: VS Code 새 파일로 `README.md` 만드는 법 + 윈도우 탐색기 확장자 숨김(`.md.txt`) 함정 한 단락 추가.
- 21.4 Publish branch: 다이얼로그 상단 Name 필드(기본값=폴더 이름 `my-keyring-game`) 언급 추가.
- 그림 마커 id·번호·순서(ch21-f01~f18), 아이콘 코너, 절 번호 전부 유지.
